# Frame in an OS kernel — a running, honest assessment

**Purpose.** A candid, evidence-based evaluation of how well Frame (the state-machine language) serves an *operating-system* codebase, written from inside the Frame OS build. Not marketing: the goal is to record where Frame genuinely earns its place, where it's neutral ceremony, and where its execution model actively fights kernel realities — so the language's value for systems work can be judged from real evidence rather than intuition.

**Method.** One entry per major milestone (B0…), each noting the Frame systems introduced, what Frame bought *over plain Rust*, and what it cost. Plus a cross-cutting "OS-level findings" section that's the real payload. **This is a living document** — appended as the project progresses; the change log at the bottom tracks revisions.

**Standing caveat.** The maintainer explicitly directed "maximize Frame utilization, even for one-state components." So some Frame usage here is *deliberate demonstration*, not the leanest engineering choice — that's called out where relevant rather than scored as a Frame win.

---

## TL;DR verdict (as of B5 Step 4e)

- **Frame is a clear net win for exactly one class of problem: genuinely state-shaped, multi-state protocols/lifecycles where the same event means different things per state.** `TcpConnection` (RFC-793) is the strongest case in the codebase; `PageFaultHandler`'s classify-then-funnel is a solid second.
- **It is roughly a wash for the many small lifecycle/mode machines** (`Mount`, `OpenFile`, `UdpSocket`, `SerialDriver`, `Scheduler`, `BlockRequest`): a `bool`/`enum` would be fewer lines; the Frame version is justified mainly by uniformity + free diagrams.
- **It does not touch the hard ~70% of a kernel** (paging, GDT/IDT, context switch, DMA, checksum/sequence arithmetic), and **every difficult bug in the project lived in native code or the Frame↔native boundary**, not in Frame state logic.
- **Its runtime model is mismatched with kernel constraints** in two concrete, load-bearing ways (per-event heap allocation; run-to-completion on a shared, stateful instance) that forced real indirection (`post`/`drain`, pending-flags) and would be unacceptable on a production hot path.
- **The project's truest value is as a Frame stress test** — it has already produced a fixed framec bug (stringified enter args) and a reusable idiom (timer-via-enter-handler + native wheel) — more than as evidence Frame makes OS development easier.

## Per-milestone

### B0 — boot (`Kernel` HSM, `SerialDriver`)
- **Bought:** the `Kernel` boot chain (`$Booting` parent → init-phase children → `$Running`) gives a readable boot sequence, and the `=> $^` panic-forwarding (write the panic path once on the parent, inherit it per phase) is a genuine, if small, win over five hand-copied panic calls. The committed `kernel.svg` is real documentation.
- **Cost / neutral:** `SerialDriver` is a 2-state init gate (`$Uninitialized → $Ready`); a `bool initialized` is equivalent and shorter. The doc admits it's "deliberately minimal." Net: the HSM proved out `=> $^` forwarding end-to-end before anything depended on it — useful as scaffolding, marginal as engineering.
- **OS note:** none of the actual B0 hard work (Limine handoff, GDT/IDT, paging bring-up, UART register programming) is Frame; it's all native. Frame organized the *narration* of boot, not boot.

### B1 — preemptive scheduling (`Scheduler`, `Task`)
- **Bought:** little, honestly. `Scheduler` is `$Idle`/`$Active` (a runnable-count mode); `Task` is a lifecycle that the architecture itself notes omits `$Running` because "on the CPU" flips every timer tick from the ISR, which **cannot fire Frame events** (non-reentrant) — so the most important scheduler state is *deliberately not* in Frame.
- **OS note (important, recurring):** the very first milestone surfaced the core mismatch — the preemptive scheduler's hot transition (which task is running) is native because the ISR can't drive Frame. Frame models the *coarse* mode; native does the real switching. This pattern repeats everywhere.

### B2 — virtual memory (`PageFaultHandler`)
- **Bought (real):** `PageFaultHandler` is the first place `=> $^` becomes load-bearing: `$Classifying → $LazyFault` recovers; on giving up, children self-send `unrecoverable()`, funneled to one parent handler that decides `$Killing` vs `$Fatal`. "What to do when we give up" is written once. In plain Rust this is a helper everyone must remember to call; the Frame version makes it structural. Classify-then-dispatch is a legitimate state-machine shape.
- **Cost:** the fault *data* (CR2 address, error code, is_user) is shared/queried state, not flowing data — correctly kept in the domain/native side. (When we later tried to "thread it" for the pipeline exercise, it didn't fit — see B5.)

### B3 — the multitasking core (`SyscallDispatcher`, `Process`, `ProcessTable`, `ElfLoader`)
- **Bought:** `SyscallDispatcher`'s `$Active.reject` funnel (the `=> $^` showcase again) is clean. `ElfLoader`'s phase pipeline with a single `$Failed` sink (rollback written once) reads well. `Process`/`ProcessTable` are reasonable lifecycle models.
- **Cost (the big one — see OS findings #2):** B3 is where Frame's execution model *created* a bug class. Implementing `exit`/`wait` the obvious way — diverging/blocking *inside* a `SyscallDispatcher` handler — corrupts the **single shared** dispatcher (leaves it stuck in `$Executing`), silently dropping the next process's syscalls (and deadlocking `wait`). The fix is pure Frame-tax: `PENDING_EXIT`/`PENDING_WAIT` flags so the block/diverge happens *after* the handler returns and the machine has settled. A plain `match` syscall demux has no "stuck mid-transition" state to corrupt.
- **OS note:** every genuinely hard B3 bug — register clobbering across the `syscall` boundary, the `IF=0` dead-task park hang, the `task_unready` scheduler race — was in native/asm or the boundary. Frame's state graphs were correct; the difficulty was elsewhere.

### B4 — block device + filesystem (`BlockRequest`, `Mount`, `OpenFile`)
- **Bought:** `OpenFile`'s "access mode is the state" (a write to a read-fd is *undispatchable*) is a nice correctness-by-construction property. The `post`/`drain` pattern is *born* here and is genuinely good design.
- **Cost / neutral:** `BlockRequest` and `Mount` are small lifecycles a `bool`/`enum` covers. The on-disk format, buffer cache, virtio-blk DMA, path walking — all native, all the actual work.
- **OS note (the decisive finding — see #3):** `post`/`drain` exists because **you cannot dispatch a Frame system from an interrupt handler**: Frame dispatch allocates (`Rc` + `BTreeMap`) through a spinlocked heap, and allocating in an ISR that preempted a lock-holding mainline is a hard deadlock. (Reentrancy is a *secondary*, mitigable concern — interrupt masking would fix it; the allocation deadlock is *not* mitigable while Frame allocates per event.) The constraint happened to push us toward the correct top-half/bottom-half architecture — a constraint that coincides with good practice.

### B5 — networking (`ArpResolver`, `RxPipeline`, `UdpSocket`, `TcpConnection`)
- **Bought (the headline win):** `TcpConnection` is the best case for Frame in the whole project. TCP's per-state event interpretation (a FIN means passive-close in `$Established`, simultaneous-close in `$FinWait1`, advance-to-`$TimeWait` in `$FinWait2`) is *exactly* what a state machine is for. The FSM was **correct on the first host run** — 16 behavioral tests mapped 1:1 to RFC-793 transitions, including simultaneous open/close, and the state logic never needed debugging. The forced enumeration of states/events produced a right-first-time control skeleton. The `$Open.rst` funnel makes RST-from-any-state one handler. The committed SVG *is* the RFC-793 diagram.
- **Bought (a genuine idiom):** `ArpResolver` established the answer to "Frame has no `after(ms)`": **arm the timer in the enter handler, cancel in the exit handler, let a native timer wheel fire a `timeout()` event via `post`/`drain`.** `TcpConnection` reuses it for retransmit + `$TimeWait`. This is a clean, reusable pattern and a real positive finding.
- **Cost #1 — the pipeline-threading was partly ceremony.** Converting `SyscallDispatcher`/`ElfLoader`/`RxPipeline` to thread data via enter params is, in straight Rust, "pass an argument." The payoff is "the data flow appears in the diagram." Worth it as a cookbook demonstration (and it surfaced the stringified-enter-arg bug); not obviously the leanest design. `PageFaultHandler` was *correctly left unconverted* because its data is shared/queried, not flowing — an honest case where the pattern doesn't fit.
- **Cost #2 — the runtime allocation mismatch is now load-bearing.** `RxPipeline.deliver()` and `TcpConnection.segment()` allocate `Rc` + `BTreeMap` **per packet/segment**. Fine for a one-connection demo at trivial rates; a non-starter for a real TCP stack (no kernel allocates on the packet path). The headline system, at production scale, would have to abandon Frame's runtime exactly where it matters most.
- **Boundary friction returned:** the live TCP work's hard parts were all native/boundary — slirp accepting host connections before the guest handshakes (forcing connection-recycling + a "wait for `[tcp] listening`" probe), and QEMU `guestfwd` connecting to its target at startup. Frame's FSM was not involved in any of these.

## OS-level findings (the real payload)

**1. Frame's leverage over a kernel is bounded by construction (~30/70).** It governs control flow; native owns data, primitives, unsafe, and arithmetic — which is the majority of a kernel and essentially all of its difficulty. So even a *great* result on the Frame portion moves the whole-project needle modestly.

**2. The Frame↔native boundary is where the bugs are.** State graphs came out correct and were cheap to test (host behavioral tests map 1:1 to transitions). Every painful bug was native (asm, DMA, sequence/checksum, slirp/QEMU quirks) or at the boundary (the shared-dispatcher corruption). Frame neither caused most of these nor helped debug them — except the shared-dispatcher class, which it *did* cause.

**3. The runtime model is the core mismatch, and it has two heads:**
   - **Per-event heap allocation.** Every dispatch allocates (`Rc<FrameEvent>` + `BTreeMap` context). **Measured (R2a, 2026-05-22):** ~**6 heap allocations per event dispatch** on the bare-metal allocator — 656 allocs across 112 `TcpConnection` dispatches (16 instances × a 7-event lifecycle), all 16 correct, no OOM on a 256 KiB heap. Consequences: (a) **cannot dispatch from an ISR** — allocating against the spinlocked heap while a preempted mainline holds the lock deadlocks; this forces `post`/`drain`. (b) **fine for control-plane, untenable on hot paths** — ~6 allocs for a connection's whole lifecycle is nothing, but ~6 allocs *per segment* on a data path is unacceptable. Both are the same root cause.
   - **Run-to-completion on a shared, stateful instance.** A handler must return so the generated `__kernel` can finish the compartment switch; the "current state" is mutable data on a single shared object. Consequence: **you cannot block or diverge inside a handler** without leaving the shared machine corrupt — forcing the `PENDING_*` deferral indirection.

**4. Where Frame is unambiguously good:** the `=> $^` parent-funnel (write a disposition once, inherit structurally) and the forced-explicitness discipline (the compiler makes you enumerate each state's handled events, which produced a correct-first-time TCP FSM). These are real, repeatable wins on state-shaped problems.

**5. Where Frame is overhead:** small 2-state lifecycle/mode machines generate ~200 lines of framework (event enum, compartments, dispatch kernel, alloc) to replace a `bool` + an `if`. Uniformity and free diagrams are the only justification; correctness/expressiveness gains are negligible at that size.

**6. Tooling/process wins are underrated:** committed, drift-checked state-graph SVGs and the 1:1 transition→test mapping are genuine workflow benefits independent of the runtime. A reviewer audits the TCP graph without reading code; CI fails on drift.

**7. framec maturity, as exercised here:** found and (maintainer-)fixed the stringified enter/exit-arg bug (typed values were round-tripped through `String`, breaking byte arrays/structs — now typed per-state context structs). A second codegen bug surfaced at B5 Step 6: a state literally named **`$Empty`** collides with framec's synthesized reserved `Empty` variant in the generated per-state context enum (used as its `Default`), producing two `Empty` enum variants → a compile error. It only triggers for that one state name; the synthesized sentinel should be namespaced against user state names. Worked around by renaming the state `$Idle` (the idiomatic name anyway), but it's a real bug to fix. Minor ongoing friction: native statements in handlers need manual semicolons; debugging a misbehaving handler means reading generated Rust. No correctness problems in the generated code itself once a system compiles.

## Scorecard

| Dimension | Verdict |
|---|---|
| Complex protocol/lifecycle FSMs (TCP, PFH) | **Net positive** — the reason to use Frame |
| Async hardware-op lifecycles (xHCI: USB enum/transfer) | **Net positive** — the TCP "enter kicks the op, completion event advances" shape generalized to register-level hardware unchanged |
| Error/disposition funneling (`=> $^`) | **Net positive** — small but repeatable |
| Forced-explicitness → correctness | **Net positive** — TCP correct first try |
| Diagrams as committed, checked docs | **Net positive** — workflow, not runtime; highest per-system payoff on register-dense protocols (USB) |
| Driver-loop dispatch by `state()` matching | **Mild negative** — when an FSM reacts to undifferentiated external completion events, the native driver must match on state names to pick the next event (stringly-typed boundary coupling) |
| Small lifecycle/mode machines | **Neutral** — ceremony vs. a `bool`/`enum` |
| Data threading via enter params | **Neutral→negative** — vs. function args; doc value only |
| Interrupt-driven paths | **Negative** — per-event alloc forces `post`/`drain` |
| Cross-core driving of an FSM (SMP) | **Net positive** — the single-owner instance + `post`/`drain` gives cross-core safety for *free* (only `Send` data crosses; no framec `Send`/`Sync` needed). The model's liability becomes the concurrency-correctness mechanism |
| Blocking/diverging control flow | **Negative** — forces `PENDING_*` deferral |
| Performance-sensitive hot paths | **Negative** — per-event allocation |
| The hard 70% (native primitives) | **N/A** — Frame doesn't apply |

## Open questions to watch (future milestones)

- **B5 scale:** many concurrent `TcpConnection` instances + a connection table — does the per-event allocation + per-instance dispatch hold up, or does it confirm the hot-path concern quantitatively?
- **B6 (USB):** ~~deep protocol HSMs with timed transitions and many ports — another genuine FSM domain; should be a *positive* case like TCP.~~ **Answered (2026-05-22):** confirmed a *positive* case — the TCP async-lifecycle pattern generalized to xHCI unchanged (see the B6 change-log entry). Caveat: the *many ports / orthogonal regions* sub-question was **not** exercised (single-port demo); it folds into B7's multi-instance-under-concurrency test.
- **B7 (SMP):** ~~the concurrency gate — does framec need `Send`+`Sync` codegen, and does the cross-core `post` story hold?~~ **Partially answered (2026-05-22):** the cross-core `post` story **holds without framec `Send`/`Sync`** — the post/drain architecture keeps the instance single-owner and only `Send` data crosses cores (see the B7 cross-core change-log entry). framec does *not* need `Send`/`Sync` codegen for the pinned-owner model. **Still open:** (a) per-CPU preemptive scheduling — does the *existing* `Scheduler`/`Process` model run cleanly per-core (native work, not a Frame question); (b) would any real design *need* a genuinely shared (multi-core-mutated) FSM instance, which would still hit the gate — so far none has.
- **A no-alloc / preallocated event path** would resolve both heads of finding #3 — worth flagging to framec as the single highest-value change for systems use.

This is the most natural data-threading use seen so far (vs. the "neutral→negative" verdict on threading generally): the loop's body genuinely *is* a state action, and the timeout escape is a first-class edge. - **2026-05-22** — **B6 (USB / xHCI) complete** — `HubPort`, `UsbEnumeration`, `UsbTransfer` over a native xHCI driver, validated on the real qemu-xhci controller + usb-kbd (controller bring-up → port reset → enumerate to `$Configured` → a real interrupt-IN HID transfer). The standout finding, and the answer to the open question "should B6 be a positive case like TCP": **the B5 networking pattern generalized cleanly to a deep hardware protocol.** All three USB systems reuse the identical shape — *the state's enter handler kicks an asynchronous hardware operation (issue a command / queue a transfer, non-blocking), and the controller's completion event, dequeued by the native driver loop, fires the FSM event that advances to the next stage.* TCP segments, xHCI command completions, and xHCI transfer events are all "an async thing finished" events; the FSM shape is the same. That this carried from a software protocol (TCP) to register-level hardware (xHCI rings/doorbells/contexts) without adaptation is a genuine, repeatable win — the strongest evidence so far that Frame's value is real on *lifecycle-shaped* problems regardless of domain. The `=> $^` funnels also recurred naturally (`HubPort.$Attached` disconnect, `UsbEnumeration.$Enumerating` fail).
  - **A nuance on run-to-completion:** finding #3 calls "can't block in a handler" a *negative* (forces `PENDING_*`). For *inherently asynchronous hardware*, the same constraint is **neutral-to-helpful** — it pushes you toward the correct driver structure (enter handler issues the op and returns; a separate loop reaps completions) rather than the tempting-but-wrong "issue and busy-wait inline." The constraint is a negative only when the *natural* code would block; for hardware that's genuinely async, blocking was never the right design.
  - **The ~30/70 split held, if anything more lopsided.** The native xHCI surface (MMIO mapping, reset choreography, DCBAA/command/event/transfer rings + cycle bits + Link-TRB wrap, slot/EP input contexts, doorbells, TRB encodings) dwarfs the Frame logic — each USB system is a 4–5-state lifecycle wrapper. Frame contributes the *control structure + the diagram*, not the protocol mechanics. For a protocol this register-dense, the committed state-graph SVGs are arguably the highest per-system payoff (a reviewer sees the enumeration/transfer lifecycle without reading 700 lines of `xhci.rs`).
  - **A new small friction (boundary, not runtime):** the native driver dispatches the right milestone event by **matching on `fsm.state().as_str()`** ("Powered"/"SlotEnabled"/…) — because different completions need different next-actions, a single generic "completed" event won't do. That's a *stringly-typed coupling* between the driver loop and the FSM's state names. It works and is localized, but it's the inverse of the usual Frame win (normally the FSM owns the dispatch; here the driver must know the FSM's stage to feed it). Worth noting as the cost of the "FSM reacts to external completion events" pattern when the events themselves are undifferentiated.
  - **Not exercised: orthogonal regions / multi-instance concurrency.** The roadmap flagged "multiple ports concurrently" as a B6 framec gate; the demo is single-port / single-device / single-flight, so that gate is **untested** — it really lands at B7 (many instances under true concurrency). Honest gap.
  - **Challenges were all native or tooling, none Frame-runtime:** the xHCI MMIO BAR sits in QEMU's high PCIe hole that Limine's HHDM doesn't map (needed the kernel's first explicit MMIO mapping); a virtio-net TX fire-and-forget buffer race surfaced when outbound IP fragmentation sent frames back-to-back; the framec `$Empty` codegen collision (now fixed in framec); and the QEMU-monitor `sendkey` needed the socket held open ~100ms so HMP processed the command. Each cost real debugging; none touched the Frame model. Consistent with finding #2 (the bugs live in native code and at the boundary, not in the state machines).
- **2026-05-22** — **B7 (SMP) Step 1: AP startup + per-CPU data — done.** Pure native, no Frame content yet (Limine MP request → `ap_entry` per AP → GS-base per-CPU block → park; `-smp 4`, all 4 cores online). Recorded here only to mark B7 begun and to flag what's coming: **the concurrency model is fine-grained per-CPU from the start** (decided with the user — per-CPU run state + per-structure locks + documented lock ordering, not a Big-Kernel-Lock staging), and the Frame-relevant reckoning is **Step 4's cross-core `post`**. The hypothesis to test there: the post/drain architecture may already give cross-core safety *without* needing framec `Send`/`Sync` codegen — if the Frame instance stays pinned to one drain context and only plain `Send` *event data* crosses cores via an MPSC queue, the non-`Send` instance never moves between cores. If that holds, the long-flagged "framec needs `Send`/`Sync`" gate is sidestepped by the architecture rather than by a codegen change; if something forces sharing the instance, we hit the gate for real (→ a framec feature request). Either outcome is a headline B7 finding. (No scorecard change at Step 1 — nothing Frame-shaped happened.)
- **2026-05-22** — **B7 Step 2: the locking foundation (`SpinLock`) — done.** Still native, no Frame content (an IRQ-safe spinlock + a 4-core counter-hammer proving no lost updates under true parallelism). Recorded because it sets up the Step-4 reckoning concretely: the lock primitive that the cross-core `post` MPSC queue will use now exists and is proven correct under contention. The interesting Frame-adjacent note is the **IRQ-safe** design — disabling interrupts while a lock is held is the *same hazard* that drove B4/B5's `post`/`drain` split (a Frame dispatch from an ISR deadlocks on the spinlocked heap). The spinlock makes that hazard explicit and bounded at the primitive level. It reinforces finding #3's first head (per-event allocation behind a spinlock is the root constraint): now there are *two* reasons an ISR mustn't dispatch a Frame system — the heap spinlock *and*, on SMP, any other lock the mainline might hold — both resolved by the same "ISRs only `post`, the owning core drains" architecture. The next step tests whether that architecture extends cleanly to cross-core posting.
- **2026-05-22** — **B7 cross-core `post` — the headline SMP finding.** The question flagged since the earliest analysis: a Frame system is *non-reentrant, allocating, and a single shared instance* — how does that survive true parallelism? Concretely: framec's generated code is **neither `Send` nor `Sync`** (it uses `Rc`, a single owned instance). The naive fear was that driving a Frame system across cores would require an `Arc`-based / `Send` framec codegen mode — a real language-runtime change. **Result: it does not. The concurrency gate is sidestepped by the post/drain architecture, with zero framec changes.** A Frame system (`EventCounter`) was driven from all 4 cores: the instance is a **local pinned to the owner core** (the BSP), other cores only enqueue plain `Send` event *data* into a `SpinLock` MPSC ring, and the owner drains + dispatches. It compiled (no `Send`/`Sync` bound ever demanded — the instance is never shared) and ran correctly (exact count of 800 = once-only dispatch of every cross-core post; a post-`close()` tick correctly dropped by `$Closed`). **The deeper point — and a genuine reframing of finding #3:** the per-event-allocation / non-reentrant / single-instance model, which is a *negative* for ISR latency and forces the `post`/`drain` indirection, turns out to be exactly what makes Frame **safe under concurrency for free**. You never have to make the state machine thread-safe; you keep it single-owner (which it already is) and marshal `Send` events to it. The `post`/`drain` pattern that was overhead at B4/B5 is the *concurrency-correctness mechanism* at B7. So the model's biggest apparent liability (shared mutable instance, no `Send`) is neutralized by the discipline the model already forces. This is the strongest single vindication of the Frame runtime model for OS work so far. **Caveats (honest):** (1) this is one instance driven by posts, not many instances migrating between cores or a Frame system *shared* (mutated) by multiple cores simultaneously — the pinned-owner model is the safe one, and it's the one the architecture naturally yields, but a design that genuinely needed a *shared* FSM instance would still hit the `Send`/`Sync` gate. (2) per-CPU preemptive scheduling (LAPIC timers, B7-1's "each core runs threads") is still ahead and is pure native work. (3) the cross-core *latency* of marshalling through a locked queue is real, same as the ISR `post`/`drain` cost — fine for control-plane events, not for a hot data path.

No Frame content; recorded to mark the milestone closed. **Stepping back across the whole project (B0–B7):** the assessment's headline conclusions held up over a full real-OS-class feature set. The repeatable Frame *wins* were consistent — complex protocol/lifecycle FSMs (PFH, TCP, USB enumeration/transfer), the `=> $^` disposition funnels, forced-explicitness producing correct-first-try state machines, drift-checked committed diagrams, and (the B7 surprise) cross-core safety for free from the single-owner + post/drain model. - **2026-05-22** — **R1a: per-core `Scheduler` Frame instances driven cross-core — done.** The first post-B7 refinement, and a direct follow-up on the cross-core-post finding: does it hold with *many* real Frame instances, not just the one `EventCounter`? Yes. Each of the 3 APs owns its own `Scheduler` instance; the BSP posts `task_ready`/`task_unready` into per-core MPSC queues; each core drains into its own Scheduler (`$Idle`→`$Active` peak 3→`$Idle`). Same shape as B7's `EventCounter`, now with N=3 instances of the *actual* B1 `Scheduler` FSM, and results read back via atomics so the BSP never touches an AP's instance. **The prediction from the B7 entry held exactly:** the single-owner + post/drain model scales to N per-core Frame schedulers with **no framec change** — the pinned-owner discipline composes. Caveat unchanged: this is the scheduling *coordination* layer (run-mode per core); per-core context-switched *execution* of multiple threads (R1b) is the deeper native piece and doesn't change the Frame conclusion. No scorecard change — it reconfirms the existing "cross-core driving of an FSM = net positive" row at higher instance count.
- **2026-05-22** — **R2a: per-event allocation finally *quantified*.** The assessment has asserted since B4 that "every Frame dispatch allocates" and called it the root constraint behind `post`/`drain` and the hot-path verdict — but never measured it. Now there's a number: a counting wrapper on the bare-metal allocator + an in-kernel stress (16 `TcpConnection` instances, each driven through a full 7-event server lifecycle = 112 dispatches) reports **656 heap allocations, ≈6 per dispatch**, with all 16 instances transitioning correctly and **no OOM** on the 256 KiB heap. So the cost is concrete: **~6 allocs/event.** Read two ways: (1) *it holds up* — N independent FSM instances at realistic event rates are fine; per-event allocation is a non-issue for control-plane lifecycles (a TCP connection is ~7 events, a USB enumeration ~5, a process lifecycle a handful). (2) *the hot-path verdict stands, now numerically* — ~6 allocations *per TCP segment* (vs per *connection*) on a data path is exactly the kind of thing the no-alloc event path (R4) must remove. This sharpens finding #3 from qualitative to measured and confirms the long-running scorecard verdicts ("hot paths = negative", "small lifecycle machines = neutral") with a concrete figure rather than an assertion. Caveat: this drives the FSMs directly (synthetic segments) to isolate the allocation/instance-scaling question; a live multi-connection TCP server with simultaneous real peers (R2b) would add network realism but not change the allocation number.
- **2026-05-22** — **R2b: N concurrent `TcpConnection` instances over real traffic.** The caveat from R2a is now closed. `tcp.rs` went from a single global connection to a real connection *table* — `Conn` slots, four listening server ports (:7–:10) plus a client slot, inbound segments resolved to a slot by 4-tuple. The key design point for the framec story: the `TcpConnection` FSM was **not changed at all**. A `CURRENT` ambient indirection points the FSM's actions (`send`, `deliver_data`, the arm/drain timers) at the resolved slot for the duration of a dispatch, so each of N instances carries its own seq state and dispatches independently without the generated code knowing the table exists. The harness opens **four simultaneous connections**, echoes on each, and the kernel reports `[tcp] served 4 connections`; the four original B5 single-connection tests still pass after the refactor (**40/40** QEMU). Two things confirmed: (1) the per-instance dispatch model holds at N live connections through the full receive→resolve→dispatch path, not just direct-drive; (2) the allocation number is **unchanged** from R2a, exactly as predicted — multiplexing happens in native table-resolution code, not in the FSM, so it adds no per-event cost. This is the multi-instance-under-real-load complement to R2a's instance-scaling measurement, and it reinforces that the right division of labor is *native owns the table/demux, Frame owns the per-connection lifecycle.*
- **2026-05-22** — **R1b: per-event allocation under N cores scheduling concurrently.** R1a gave each core a `Scheduler` Frame instance but only *tracked* counts via cross-core posts; no core actually time-sliced anything. R1b closes that: each AP now runs a real per-core run queue (`pcsched.rs`) — it spawns kernel-thread workers and time-slices them under its own LAPIC timer, which became a full-frame context-switching ISR (`isr_lapic_timer` → `lapic_schedule` → per-core `pcsched::schedule`, pure native). Spawning/exiting each worker drives that core's own `Scheduler` instance through $Idle→$Active→$Idle, every dispatch in a per-core interrupts-off critical section — the *same* native/Frame discipline as the BSP's single-core `sched.rs`, now replicated N-fold. This is the load case R1a couldn't reach: **3 cores each running a live `Scheduler` instance through real preemptive execution simultaneously, all allocating per dispatch against the one shared, spin-locked heap.** Result: every AP ran all 3 workers to completion with **~42 context switches** and ended `$Idle`; the BSP measured **~57 heap allocs** for the whole phase (3 cores × 6 dispatches ≈ **3.2 allocs/dispatch**). Two findings: (1) **the per-event-allocation model holds under concurrent multi-core scheduling** — no corruption, no deadlock, every instance correct, confirming the B7 pinned-owner verdict (framec needs no `Send`/`Sync`) extends from cross-core *posting* to N cores each *running* their own FSM under preemption. (2) **the per-dispatch alloc cost tracks event complexity** — ~3.2 allocs for the parameterless `Scheduler` events here vs R2a's ~6 for the param-carrying `TcpConnection.segment(seg)`: the gap is the per-parameter Vec/map allocation, which sharpens finding #3 (the no-alloc path R4 must target the *parameter marshalling*, not just the event/context objects). The hard discipline that makes all of this safe is unchanged from B1: the timer ISR is **pure native** (it never dispatches Frame — that would re-enter the allocating, non-reentrant system from interrupt context and deadlock on the heap lock); Frame dispatch happens only in thread context with interrupts off. R1b just proved that discipline composes across cores. **Scope:** kernel threads (ring 0); per-core user processes (ring-3-on-APs + per-CPU TSS.RSP0) are deferred to R5 and aren't needed for the Frame question. Validated by `smp_pcsched_r1b` (**41/41** QEMU).
- **2026-05-22** — **Analysis: is Frame's single-thread-per-instance model a safety advantage?** (Prompted by reflection on the B7/R1 results — recorded because it's the load-bearing argument behind the whole OS architecture.) "Can Frame be multi-threaded?" has two answers because it's two questions. **(1) Many instances, one per thread/core — yes, proven.** R1a/R1b/R2b drove N independent FSM instances across N cores concurrently with zero locks on FSM state and zero races; it's embarrassingly parallel. **(2) One instance mutated concurrently by many threads — no, and deliberately.** framec's generated code is non-`Send`/non-`Sync`; a single instance cannot be driven by multiple threads at once, and you can't recover the capability without destroying the thing — a `Mutex<T>` wrapper serializes every dispatch (lock contention, *no* parallelism, plus reentrancy-deadlock risk if a handler dispatches into itself). The reason isn't a missing feature: **a state machine and shared-memory concurrency are semantically incompatible.** Frame's execution model is run-to-completion (RTC) — an event, including its transition and enter/exit handlers, is fully processed before the next starts — and RTC is precisely what makes "what state am I in?" answerable. Two threads dispatching into one instance produce torn transitions, an exit handler racing the next enter handler, a half-mutated compartment observed mid-flight: the abstraction collapses. This is the actor model (the Erlang backend literally is `gen_statem`: concurrency *between* actors, never *within* one). **Verdict — net safety advantage, with one trap.** It makes an entire bug class *unrepresentable*: data races on state, reentrant-dispatch corruption, torn transitions. Rust's `Send`/`Sync` and Frame's RTC jointly *forbid* them — you cannot even move a non-`Send` instance across a thread boundary without `unsafe`. The recurring "does framec need `Send`/`Sync`?" gate answered itself across B7/R1: **no — because no real systems problem ever needed a shared instance.** Every one decomposed cleanly into per-instance ownership (post/drain for ISRs, single-owner-per-core for SMP, the connection table for TCP). That the constraint never bound in three milestones of systems code is evidence it's the *right* constraint. The deeper, language-design point: RFC-0026's preservation theorem only exists because there's a well-defined operational step (one RTC dispatch) to preserve an invariant across; concurrent intra-instance mutation means there is no atomic "step," so single-ownership is what lets you even *state* soundness. **The trap:** the guarantee only holds if you honor it by message-passing (Send data crosses, instances don't). The moment someone "fixes" the limitation with a `Mutex`, they trade a compile-time guarantee for a runtime lock and get the worst of both. So the honest framing: Frame **trades a capability you almost never want (shared-instance concurrency) for a guarantee you always want (race-free state)**, and the cost only appears if a workload genuinely cannot be sharded — which, so far, has never happened.
- **2026-05-22** — **R3a: the orthogonal-regions question, answered on real hardware.** B6 was single-port/single-device and left open the *orthogonal regions* sub-question: many concurrent lifecycle FSMs of the same type, managed as a group. R3a closes it for USB. `xhci.rs` went from single-flight globals to a per-device table (`Device` slots + a `CUR_DEV` ambient index — the exact `tcp.rs` connection-table pattern), and the *unchanged* `HubPort` / `UsbEnumeration` / `UsbTransfer` FSMs now operate on "the current device." Two HID devices (`usb-kbd` port 5, `usb-mouse` port 6) are enumerated **concurrently**: one `HubPort` + one `UsbEnumeration` instance per device coexist, and a single driver loop demuxes each xHCI completion to the right instance by slot, pointing `CUR_DEV` at it for the dispatch — `slot 1 enabled → addressed → slot 2 enabled → addressed → configured(1) → configured(2) → enumerated 2 of 2`. The significance over R2a/R2b (which also ran N same-type instances): this is driven by **real asynchronous hardware completion events demuxed by an external key (the xHCI slot)**, not synthetic segments — the USB analog of the TCP connection table, on hardware. Same finding, reinforced from a third subsystem: *native owns the table/demux, Frame owns each lifecycle*, and the per-instance model holds for concurrent same-type FSMs regardless of whether the events come from a synthetic harness, a live network peer, or a hardware controller's event ring. One honest seam worth recording: slot assignment **must** be serialized (xHCI's Enable Slot completion carries no port, so you can't correlate a returned slot to a requesting device unless only one is outstanding) — a hardware-imposed serialization point, not a Frame limitation; everything after Address Device carries the slot and interleaves freely. Validated by `usb_multiport_r3a`; the four B6 single-device tests still pass (the keyboard stays device 0 / slot 1, so the B6-4 keypress transfer is unaffected). The B6 → R3a path is the same regression-safe table refactor used for R2b: migrate globals → table with one ambient `CUR_DEV`, prove behavior-identical on the single-instance suite, *then* turn on N.
- **2026-05-22** — **R3b: a new device class + transfer type (USB mass storage / SCSI over bulk).** R3a stayed within one device class (HID); R3b adds a genuinely different one — a `usb-storage` device, bulk endpoints (not the HID interrupt-IN), and a layered protocol (SCSI inside Bulk-Only Transport inside USB bulk). Two things this surfaced are worth recording. **(1) Identity must come from the descriptor, not the table.** Adding a USB3 storage device immediately broke the B6 assumption "device 0 is the keyboard" — being SuperSpeed it sorts onto a lower root-hub port than the USB2 HID devices, so it became device 0 and the keypress transfer started addressing the wrong device. The fix is what a real OS does: read each device's *configuration descriptor* after enumeration, record the interface class/protocol + bulk endpoint addresses, and route by class (keypress → HID keyboard, SCSI → mass storage). This is a native concern entirely; Frame neither helped nor hindered — it's pointer-walking a descriptor blob, the same ~70% of a kernel that's never Frame-shaped. **(2) BOT is a clean Frame win, for the same reason TCP was.** The Bulk-Only Transport transaction is a literal three-phase protocol — Command (CBW out) → Data (in) → Status (CSW in) — where each phase is a separate async bulk transfer whose completion advances the machine. That is exactly the `enter-handler-kicks-the-transfer / completion-event-advances-the-FSM` shape, now with three phases instead of one, and the `fail()`-from-any-phase funnel is the `TcpConnection.$Open` / `=> $^` idiom again. The `UsbMsd` FSM came out clean and was correct on the first QEMU run (INQUIRY, READ CAPACITY(10), and READ(10) of block 0 all returned correct data — vendor "QEMU", 128×512-byte blocks, and the "FRAMEOS!" magic). The pattern holds across yet another protocol: native owns the byte layout (CBW/CSW/CDB) and the rings; Frame owns the protocol phase lifecycle. Net for R3: the orthogonal-regions question is fully answered (R3a, concurrent same-class; R3b, a second class with its own transfer type), and nothing new about Frame's *limits* appeared — the costs are the same per-event-allocation and ISR-can't-dispatch facts already on the scorecard; the wins are the same "multi-state protocol with per-state event meaning" sweet spot. Validated by `usb_msd_r3b` (QEMU) + `usb_msd_behavior` (4 host tests) + a state-graph snapshot.
- **2026-05-22** — **R7a: scheduler-as-actor — the message-passing principle applied *inside* a core.** The multithreading analysis (above) named the rule "post across contexts, call within a context," and flagged one place the OS violated its own discipline: R1a (`ksched.rs`) drove the per-core `Scheduler` through a mailbox, but R1b (`pcsched.rs`) drove the *same FSM* synchronously inside a lock (`with_sched` + `without_interrupts`), because two contexts on the core touched it — a worker (in `exit_current`) and the idle loop. R7a removes that split: a per-core mailbox now carries `Ready`/`Unready` messages; workers and the spawn path only *post*, and the idle loop is the **sole** drainer that dispatches to the `Scheduler` and reads `is_idle()`. One owner ⇒ no lock around the FSM. The `.frs` is unchanged — only *how the instance is driven* changed, from shared-lock to single-owner mailbox. Three things worth recording. **(1) It's behavior-identical** (`smp_pcsched_r1b`: 3 threads sliced, ~40 switches, ended idle, 57 allocs — same numbers as before), confirming this is a pure coupling refactor, not a semantic change. **(2) The one residual critical section is honest.** `exit_current` still wraps the Dead-mark + the mailbox push in `without_interrupts` — but that now guards a *native* handoff (don't let a timer tick strand the worker between marking itself dead and posting its retire), not an FSM dispatch. The FSM itself is lock-free. **(3) This is the native hand-rolling of RFC-0038.** The worker "casts" `task_unready` to the Scheduler's mailbox and the executor (idle loop) delivers it; the missing framec primitive (`@@[cast]` + a drain hook) would replace the mailbox/`SchedMsg`/drain boilerplate with language syntax. R7a is the strongest dogfooding evidence for that RFC so far: it's the *third* hand-rolled instance of post/drain (after ISR→mainline and core→core), now applied to defeat an in-core shared lock — exactly the deferred-dispatch use the RFC argues the runtime spec is missing.
- **2026-05-22** — **R7b/R7c: one mailbox primitive, and a finding about where message-passing actually lives.** R7a left three byte-for-byte-identical hand-rolled FIFO rings (`crosscore::PostQueue`, `ksched::SchedQueue`, `pcsched::SchedMailbox`). R7b extracts one generic `reactor::Mailbox<T, CAP>` (fixed-capacity, no-alloc, `const`-constructible) and routes all three through it — the native realization of RFC-0038's standard mailbox, behavior-preserving (the SMP suite is unchanged). The more useful outcome was R7c's **negative result**, worth recording because it sharpens the whole message-passing thesis: the roadmap had assumed block/net/USB would each get a software mailbox "to match net," but investigation showed **none of them has a software event queue** — net's `RxPipeline`, B4 block, and USB all drain the **hardware** ring directly (virtio used-ring + an `IRQ_PENDING` wakeup; the xHCI event ring via poll). The hardware ring *is* the mailbox. So the OS actually has exactly *two* kinds of post/drain: (a) **hardware→software**, where the queue is a DMA ring the device fills and the driver drains (block, net, USB) — no software mailbox is correct there, and bolting one on would duplicate the ring; and (b) **software→software**, cross-context posts between code paths that can't dispatch directly (ISR→mainline deferred events, core→core, in-core scheduler) — *those* are the three `Mailbox` users, now unified. The lesson for framec's RFC-0038: a language-level deferred-send / `@@[cast]` primitive is the right abstraction for case (b) — the software mailboxes the kernel hand-rolls — but it should **not** try to subsume case (a); device DMA rings are their own (hardware-defined) transport that native code will always own. The cleanest division stays: hardware rings are native; the Frame-level mailbox is for software-to-software deferred dispatch. R7b validated by the SMP suite; no functional change in R7c.
- **2026-05-22** — **R5: deeper SMP correctness — pure native, and that's the point.** R5 added a lock-ordering rank checker on `SpinLock` (panic-on-reversal, with a cross-core nested-lock A→B stress) and per-CPU TSS + double-fault IST stacks (each core `ltr`s its own TSS; #DF → IST1, so a fault lands on a known-good per-core stack instead of triple-faulting). Recorded here only to mark the boundary: **none of this is Frame.** It's register/segment/descriptor and lock mechanics — the same ~70% of a kernel the assessment has said all along Frame doesn't touch. There was no FSM to write, no `.frs`, no Frame finding; the `Scheduler`/`Process`/etc. systems are untouched. This is worth stating positively rather than as an omission: a language for *control-flow state machines* correctly has nothing to say about lock hierarchies or TSS layout, and R5 is a clean example of the division of labor the whole project has mapped — Frame owns protocol/lifecycle state; native owns the machine. (Sleep-locks, the one R5 piece with a plausible Frame angle — a blocking primitive could be modeled as a wait/wake lifecycle — were deferred because they need scheduler integration the current AP model lacks; if revisited they'd be the only part of R5 where a Frame system might earn its place.) Validated by `smp_nested_lock_r5` + `smp_percpu_tss_r5`; the B3 ring-3/syscall path (which depends on the reworked GDT/TSS) still passes.
- **2026-05-22** — **B8: an interactive shell — Frame's sweet spot meets the kernel's hardest bugs.** The first V1.0-track step: a typeable console. The new native plumbing (serial RX + a `read_line` syscall + a console line discipline) is the usual ~70% — no Frame. But the shell itself, `ish`, **reuses the `Parser` FSM** (the same `parser.frs` the hosted shell compiles) to tokenize input in ring 3 — the "one source, two targets" property again, and a REPL is a textbook small state machine. The headline lesson, though, is about the *runtime model under real concurrency*: getting "type a command → a program runs → back to the prompt" working required `fork`, a blocking `read_line`, blocking disk I/O, and `wait()` to all compose — and that exposed a genuine **blocking-I/O bug** (virtio-blk reads busy-`hlt`'d for a completion IRQ instead of yielding; fixed to block the process + wake on the IRQ via `sched::wake_pid`, which is also what lets a *forked* child's disk read complete — it yields the CPU so the IRQ can be delivered and serviced). Two further bugs surfaced only when the shell ran as a lean, demos-skipped *interactive* build, and both were instructive about **latent coupling in the native substrate, not Frame**: (1) the interactive kernel re-ran the entire B0–B7 self-test suite (billion-iteration SMP spin loops) before reaching the prompt — far too slow under TCG — so an interactive build now boots *straight to a shell*, gating the self-tests to the default build; (2) a `#PF` at address 0 on the first context switch into a ring-3 process traced to `gdt::set_rsp0` reading this core's index via `gs:[0]` while the **BSP's per-CPU GS base was still zero** — its initialization (`percpu::init_this_cpu`) had been buried *inside the SMP demo block* the interactive build skips. The fix makes BSP per-CPU init explicit on the interactive path. Both are exactly the standing finding in miniature: **Frame models the lifecycle cleanly (the REPL, the parser), but the kernel's genuinely hard problems — interrupt delivery, per-CPU register state, scheduler/stack discipline — live entirely in the native layer Frame doesn't touch.** Shipped working *and verified end-to-end*: typing `/bin/hello` at the `frameos$` prompt forks, the child reads the ELF **from disk**, execs it, prints, and the shell survives to the next prompt — then `exit` halts cleanly (`cargo xtask console-test` green; default 45-suite unaffected — the feature is gated). This is the disk-loaded-from-shell path that gates B11/B12 (on-device toolchain).
- **2026-05-22** — **B9-1: a growable heap (`brk`) — pure-native mechanism, and honestly so.** The first B9 step toward hosting real toolchains, which need megabytes of heap rather than the fixed 64 KiB static the user programs shipped with. `brk(new_end)` (syscall #10) gives a process a heap it can grow on demand: the kernel demand-maps fresh zeroed `USER|WRITABLE` pages over `[old_break, new_break)` into the process's *own* address space (and unmaps + frees on shrink). The break is per-process state on the `Tcb` (`heap_brk`), seeded at a dedicated VA region (`USER_HEAP_BASE = 0x3000_0000`) chosen to never collide with the program image (`0x1000_0000`) or the user stack (`0x2000_0000`); it composes correctly with the existing process primitives — a `fork`ed child inherits the parent's break (the heap pages ride along in the copied user half), `exec` resets it (new image ⇒ empty heap), and reap's `free_address_space` reclaims the pages with the rest of the user space, so no separate bookkeeping was needed. Recorded here mainly to mark the division of labor again: **there is no Frame system in B9-1, and there shouldn't be.** A program break is a single per-process pointer and a map/unmap loop — address-space mechanism, the same native ~70% as the frame allocator and page tables. There's no lifecycle, no protocol, no per-state event meaning for an FSM to model; inventing one would be ceremony. Frame's place in the V1.0 toolchain track is still ahead and specific (the **build driver** — `framec → mrustc → tcc → link` as a pipeline FSM with a `$Failed` sink). Validated by `brktest` (grows its heap 1 MiB via `brk`, writes a distinct value to every u64 across the new megabyte and reads it all back) + the `brk_growable_heap_b9` smoke test; the default suite is unaffected.
- **2026-05-22** — **B9-2: `argv` through `exec` — the shell finally hands programs their arguments.** `exec_argv` (syscall #11) carries a packed argv (`argc` NUL-terminated strings, `argv[0]` = the program path) from ring 3 into the kernel, which loads the ELF and then writes a **System V x86-64 initial stack** — `argc`, the `argv[]` pointer array, a NULL, an empty `envp` NULL, an `auxv` AT_NULL terminator, and the argument string bytes copied to the top of the page — onto the new program's stack, entering it with `rsp` pointed at `argc`. The shell (`ish`) now packs its parsed tokens into that buffer (so `/bin/argtest alpha beta` delivers three argv entries), and `argtest` reads them back. Two implementation notes worth keeping. **(1) The stack build is trivially correct *because* of an earlier decision.** `exec_image` already switches CR3 to the new address space (`exec_into`) before setting the trap frame — so by the time the kernel lays the argv structure down, the new program's user stack is simply *mapped in the current CR3*, and the kernel writes user VAs directly (ring 0 writing a present, writable user page) rather than juggling the new space's physical frames through the HHDM. The ABI layout (16-aligned `rsp` at `argc`, strings above the pointer block) is the only fiddly part, and it's pure native byte-laying. **(2) The crt0 gap is real and native.** A Rust `extern "C" fn _start` cannot see the initial stack — the args live *at* `rsp`, not in registers — so `argtest`'s `_start` is a three-instruction asm shim (`mov rdi, rsp; and rsp, -16; call argtest_main`) that hands the entry stack pointer to Rust. This is exactly the crt0 a libc provides, and it foreshadows B10: `frame-libc` will own this shim (and `envp`/`auxv` parsing) for C programs. As with B9-1, **there is no Frame system here and shouldn't be** — argv marshalling is calling-convention plumbing, the native ~70%. Frame's place in the toolchain track remains the build *driver* (B11+). Validated by `console-test` (the shell types `/bin/argtest alpha beta`; the program echoes `argv[1]=alpha`, `argv[2]=beta`); the default 46-test suite is unaffected (the existing no-arg `exec`/spawn path is untouched — argv rides a new syscall).
- **2026-05-23** — **B9-3: the file write path — and a small but real Frame win hiding in it.** The fd API was read-only (`open`/`read`/`close`); B9-3 adds what a libc/toolchain needs to *produce* output and inspect inputs: `open` gained a read/write flag (back-compatibly — old callers pass 0 = read), and new syscalls `write` (to a file at the fd's offset), `lseek` (SET/CUR/END), `fstat`/`stat` (size), and `dup` (a shared-offset descriptor). The new on-disk primitive is `fs::write_at` — a random-access write that allocates direct blocks on demand and grows the inode size — which is the write twin of the existing `read_at`; everything else is fd-table bookkeeping (offsets, the writable flag, slot duplication). Mostly this is the native ~70% again. **But there's one genuine Frame contribution worth recording, because it's the first time the `OpenFile` FSM has *earned its keep* rather than just existing:** the access-mode gate. `write()` calls `slot.file.write()` and bails unless the instance `is_writing()`; `read()` does the same for `is_reading()`. So a descriptor opened read-only *cannot* write and vice-versa — not by an `if` the C code has to remember at every call site, but because the `OpenFile` state machine (`$Reading` / `$Writing` / `$Closed`) makes the wrong-mode event a no-op by construction, and a closed fd is inert in the same way. Back at B4 the `OpenFile` system looked like ceremony (one FS, trivial dispatch); B9-3 is where the modeled access mode actually does work, gating a real read/write/dup matrix. It's a modest example, but it's the honest kind: Frame is carrying the *mode-correctness invariant* while native carries the bytes. `getcwd` and `envp` were deferred — both need state (a per-process cwd; an environment) that belongs with `frame-libc` (B10), which is the consumer that will define their shape. Validated by `fwtest` (create `/tmp.txt`, write, seek-and-overwrite the middle via `write_at`, `fstat`, reopen and verify, including a `dup`'d fd that shares the offset) + the `file_write_roundtrip_b9` smoke test; the rest of the suite is unaffected.
- **2026-05-23** — **B9.5: a toolchain-ready filesystem — pure native, and a clean "geometric capacity from linear code" lesson.** B10's first libc-linked program (`cmain`, 8.5 KB) immediately exceeded the FS's **7 KB** file cap (14 direct blocks × 512), and the inode was exactly full at 64 bytes — so the limit was real, not tunable. Rather than a constant fudge, the FS got the proper scalable format: a 128-byte inode with **28 direct + single-indirect + double-indirect** block pointers (max file ~8 MiB), and a **multi-block free-block bitmap** derived from disk size (`Layout::for_total(total_blocks)`, addressing ~2 TB). The kernel maps a file block index through `block_for` (direct → single → double, lazily allocating the index blocks). Worth recording for the *design* lesson, not a Frame one (there's no FSM here — block maps and bitmaps are mechanism): this is the textbook Unix inode, and its virtue is that **each indirect tier is one more bounded `if`/walk while capacity grows geometrically (28 → +128 → +128² blocks)** — linear code, exponential reach. The same is true of the bitmap (one bit per block, indexed by arithmetic) and inode table (more blocks, same `inode_loc` division). The format is shared (`frame_os_shared::fs`) so the kernel driver and host `mkfs` can't drift; the host writes direct-only for now (staged binaries are ≤14 KB; tcc-scale staging teaches the host indirect at B11). The default test disk went 1 MiB → 4 MiB to actually exercise the multi-block bitmap; the *format* scales far past that, but the host `mkfs` builds the whole image in RAM and each smoke test copies it, so the test image stays modest by choice — a deliberate split between *format capability* and *test cost*. Validated by a 128 KiB (double-indirect) write/verify round-trip folded into `fs_file_roundtrip_b4`; the full FS/VFS/shell/persist suite still passes on the new format. The honest header: B9 + B9.5 are ~600 lines of native FS/syscall/address-space mechanism with two small Frame touches (the `OpenFile` mode gate, B9-3) — the toolchain substrate is overwhelmingly the native ~70%, exactly as the scorecard predicts; Frame's turn comes at the build *driver*.
