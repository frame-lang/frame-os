# Frame in an OS kernel â€” a running, honest assessment

**Purpose.** A candid, evidence-based evaluation of how well Frame (the state-machine language) serves an *operating-system* codebase, written from inside the Frame OS build. Not marketing: the goal is to record where Frame genuinely earns its place, where it's neutral ceremony, and where its execution model actively fights kernel realities â€” so the language's value for systems work can be judged from real evidence rather than intuition.

**Method.** One entry per major milestone (B0â€¦), each noting the Frame systems introduced, what Frame bought *over plain Rust*, and what it cost. Plus a cross-cutting "OS-level findings" section that's the real payload. **This is a living document** â€” appended as the project progresses; the change log at the bottom tracks revisions.

**Standing caveat.** The maintainer explicitly directed "maximize Frame utilization, even for one-state components." So some Frame usage here is *deliberate demonstration*, not the leanest engineering choice â€” that's called out where relevant rather than scored as a Frame win.

---

## TL;DR verdict (as of B5 Step 4e)

- **Frame is a clear net win for exactly one class of problem: genuinely state-shaped, multi-state protocols/lifecycles where the same event means different things per state.** `TcpConnection` (RFC-793) is the strongest case in the codebase; `PageFaultHandler`'s classify-then-funnel is a solid second.
- **It is roughly a wash for the many small lifecycle/mode machines** (`Mount`, `OpenFile`, `UdpSocket`, `SerialDriver`, `Scheduler`, `BlockRequest`): a `bool`/`enum` would be fewer lines; the Frame version is justified mainly by uniformity + free diagrams.
- **It does not touch the hard ~70% of a kernel** (paging, GDT/IDT, context switch, DMA, checksum/sequence arithmetic), and **every difficult bug in the project lived in native code or the Frameâ†”native boundary**, not in Frame state logic.
- **Its runtime model is mismatched with kernel constraints** in two concrete, load-bearing ways (per-event heap allocation; run-to-completion on a shared, stateful instance) that forced real indirection (`post`/`drain`, pending-flags) and would be unacceptable on a production hot path.
- **The project's truest value is as a Frame stress test** â€” it has already produced a fixed framec bug (stringified enter args) and a reusable idiom (timer-via-enter-handler + native wheel) â€” more than as evidence Frame makes OS development easier.

## Per-milestone

### B0 â€” boot (`Kernel` HSM, `SerialDriver`)
- **Bought:** the `Kernel` boot chain (`$Booting` parent â†’ init-phase children â†’ `$Running`) gives a readable boot sequence, and the `=> $^` panic-forwarding (write the panic path once on the parent, inherit it per phase) is a genuine, if small, win over five hand-copied panic calls. The committed `kernel.svg` is real documentation.
- **Cost / neutral:** `SerialDriver` is a 2-state init gate (`$Uninitialized â†’ $Ready`); a `bool initialized` is equivalent and shorter. The doc admits it's "deliberately minimal." Net: the HSM proved out `=> $^` forwarding end-to-end before anything depended on it â€” useful as scaffolding, marginal as engineering.
- **OS note:** none of the actual B0 hard work (Limine handoff, GDT/IDT, paging bring-up, UART register programming) is Frame; it's all native. Frame organized the *narration* of boot, not boot.

### B1 â€” preemptive scheduling (`Scheduler`, `Task`)
- **Bought:** little, honestly. `Scheduler` is `$Idle`/`$Active` (a runnable-count mode); `Task` is a lifecycle that the architecture itself notes omits `$Running` because "on the CPU" flips every timer tick from the ISR, which **cannot fire Frame events** (non-reentrant) â€” so the most important scheduler state is *deliberately not* in Frame.
- **OS note (important, recurring):** the very first milestone surfaced the core mismatch â€” the preemptive scheduler's hot transition (which task is running) is native because the ISR can't drive Frame. Frame models the *coarse* mode; native does the real switching. This pattern repeats everywhere.

### B2 â€” virtual memory (`PageFaultHandler`)
- **Bought (real):** `PageFaultHandler` is the first place `=> $^` becomes load-bearing: `$Classifying â†’ $LazyFault` recovers; on giving up, children self-send `unrecoverable()`, funneled to one parent handler that decides `$Killing` vs `$Fatal`. "What to do when we give up" is written once. In plain Rust this is a helper everyone must remember to call; the Frame version makes it structural. Classify-then-dispatch is a legitimate state-machine shape.
- **Cost:** the fault *data* (CR2 address, error code, is_user) is shared/queried state, not flowing data â€” correctly kept in the domain/native side. (When we later tried to "thread it" for the pipeline exercise, it didn't fit â€” see B5.)

### B3 â€” the multitasking core (`SyscallDispatcher`, `Process`, `ProcessTable`, `ElfLoader`)
- **Bought:** `SyscallDispatcher`'s `$Active.reject` funnel (the `=> $^` showcase again) is clean. `ElfLoader`'s phase pipeline with a single `$Failed` sink (rollback written once) reads well. `Process`/`ProcessTable` are reasonable lifecycle models.
- **Cost (the big one â€” see OS findings #2):** B3 is where Frame's execution model *created* a bug class. Implementing `exit`/`wait` the obvious way â€” diverging/blocking *inside* a `SyscallDispatcher` handler â€” corrupts the **single shared** dispatcher (leaves it stuck in `$Executing`), silently dropping the next process's syscalls (and deadlocking `wait`). The fix is pure Frame-tax: `PENDING_EXIT`/`PENDING_WAIT` flags so the block/diverge happens *after* the handler returns and the machine has settled. A plain `match` syscall demux has no "stuck mid-transition" state to corrupt.
- **OS note:** every genuinely hard B3 bug â€” register clobbering across the `syscall` boundary, the `IF=0` dead-task park hang, the `task_unready` scheduler race â€” was in native/asm or the boundary. Frame's state graphs were correct; the difficulty was elsewhere.

### B4 â€” block device + filesystem (`BlockRequest`, `Mount`, `OpenFile`)
- **Bought:** `OpenFile`'s "access mode is the state" (a write to a read-fd is *undispatchable*) is a nice correctness-by-construction property. The `post`/`drain` pattern is *born* here and is genuinely good design.
- **Cost / neutral:** `BlockRequest` and `Mount` are small lifecycles a `bool`/`enum` covers. The on-disk format, buffer cache, virtio-blk DMA, path walking â€” all native, all the actual work.
- **OS note (the decisive finding â€” see #3):** `post`/`drain` exists because **you cannot dispatch a Frame system from an interrupt handler**: Frame dispatch allocates (`Rc` + `BTreeMap`) through a spinlocked heap, and allocating in an ISR that preempted a lock-holding mainline is a hard deadlock. (Reentrancy is a *secondary*, mitigable concern â€” interrupt masking would fix it; the allocation deadlock is *not* mitigable while Frame allocates per event.) The constraint happened to push us toward the correct top-half/bottom-half architecture â€” a constraint that coincides with good practice.

### B5 â€” networking (`ArpResolver`, `RxPipeline`, `UdpSocket`, `TcpConnection`)
- **Bought (the headline win):** `TcpConnection` is the best case for Frame in the whole project. TCP's per-state event interpretation (a FIN means passive-close in `$Established`, simultaneous-close in `$FinWait1`, advance-to-`$TimeWait` in `$FinWait2`) is *exactly* what a state machine is for. The FSM was **correct on the first host run** â€” 16 behavioral tests mapped 1:1 to RFC-793 transitions, including simultaneous open/close, and the state logic never needed debugging. The forced enumeration of states/events produced a right-first-time control skeleton. The `$Open.rst` funnel makes RST-from-any-state one handler. The committed SVG *is* the RFC-793 diagram.
- **Bought (a genuine idiom):** `ArpResolver` established the answer to "Frame has no `after(ms)`": **arm the timer in the enter handler, cancel in the exit handler, let a native timer wheel fire a `timeout()` event via `post`/`drain`.** `TcpConnection` reuses it for retransmit + `$TimeWait`. This is a clean, reusable pattern and a real positive finding.
- **Cost #1 â€” the pipeline-threading was partly ceremony.** Converting `SyscallDispatcher`/`ElfLoader`/`RxPipeline` to thread data via enter params is, in straight Rust, "pass an argument." The payoff is "the data flow appears in the diagram." Worth it as a cookbook demonstration (and it surfaced the stringified-enter-arg bug); not obviously the leanest design. `PageFaultHandler` was *correctly left unconverted* because its data is shared/queried, not flowing â€” an honest case where the pattern doesn't fit.
- **Cost #2 â€” the runtime allocation mismatch is now load-bearing.** `RxPipeline.deliver()` and `TcpConnection.segment()` allocate `Rc` + `BTreeMap` **per packet/segment**. Fine for a one-connection demo at trivial rates; a non-starter for a real TCP stack (no kernel allocates on the packet path). The headline system, at production scale, would have to abandon Frame's runtime exactly where it matters most.
- **Boundary friction returned:** the live TCP work's hard parts were all native/boundary â€” slirp accepting host connections before the guest handshakes (forcing connection-recycling + a "wait for `[tcp] listening`" probe), and QEMU `guestfwd` connecting to its target at startup. Frame's FSM was not involved in any of these.

## OS-level findings (the real payload)

**1. Frame's leverage over a kernel is bounded by construction (~30/70).** It governs control flow; native owns data, primitives, unsafe, and arithmetic â€” which is the majority of a kernel and essentially all of its difficulty. So even a *great* result on the Frame portion moves the whole-project needle modestly.

**2. The Frameâ†”native boundary is where the bugs are.** State graphs came out correct and were cheap to test (host behavioral tests map 1:1 to transitions). Every painful bug was native (asm, DMA, sequence/checksum, slirp/QEMU quirks) or at the boundary (the shared-dispatcher corruption). Frame neither caused most of these nor helped debug them â€” except the shared-dispatcher class, which it *did* cause.

**3. The runtime model is the core mismatch, and it has two heads:**
   - **Per-event heap allocation.** Every dispatch allocates (`Rc<FrameEvent>` + `BTreeMap` context). **Measured (R2a, 2026-05-22):** ~**6 heap allocations per event dispatch** on the bare-metal allocator â€” 656 allocs across 112 `TcpConnection` dispatches (16 instances Ã— a 7-event lifecycle), all 16 correct, no OOM on a 256 KiB heap. Consequences: (a) **cannot dispatch from an ISR** â€” allocating against the spinlocked heap while a preempted mainline holds the lock deadlocks; this forces `post`/`drain`. (b) **fine for control-plane, untenable on hot paths** â€” ~6 allocs for a connection's whole lifecycle is nothing, but ~6 allocs *per segment* on a data path is unacceptable. Both are the same root cause.
   - **Run-to-completion on a shared, stateful instance.** A handler must return so the generated `__kernel` can finish the compartment switch; the "current state" is mutable data on a single shared object. Consequence: **you cannot block or diverge inside a handler** without leaving the shared machine corrupt â€” forcing the `PENDING_*` deferral indirection.

**4. Where Frame is unambiguously good:** the `=> $^` parent-funnel (write a disposition once, inherit structurally) and the forced-explicitness discipline (the compiler makes you enumerate each state's handled events, which produced a correct-first-time TCP FSM). These are real, repeatable wins on state-shaped problems.

**5. Where Frame is overhead:** small 2-state lifecycle/mode machines generate ~200 lines of framework (event enum, compartments, dispatch kernel, alloc) to replace a `bool` + an `if`. Uniformity and free diagrams are the only justification; correctness/expressiveness gains are negligible at that size.

**6. Tooling/process wins are underrated:** committed, drift-checked state-graph SVGs and the 1:1 transitionâ†’test mapping are genuine workflow benefits independent of the runtime. A reviewer audits the TCP graph without reading code; CI fails on drift.

**7. framec maturity, as exercised here:** found and (maintainer-)fixed the stringified enter/exit-arg bug (typed values were round-tripped through `String`, breaking byte arrays/structs â€” now typed per-state context structs). A second codegen bug surfaced at B5 Step 6: a state literally named **`$Empty`** collides with framec's synthesized reserved `Empty` variant in the generated per-state context enum (used as its `Default`), producing two `Empty` enum variants â†’ a compile error. It only triggers for that one state name; the synthesized sentinel should be namespaced against user state names. Worked around by renaming the state `$Idle` (the idiomatic name anyway), but it's a real bug to fix. Minor ongoing friction: native statements in handlers need manual semicolons; debugging a misbehaving handler means reading generated Rust. No correctness problems in the generated code itself once a system compiles.

## Scorecard

| Dimension | Verdict |
|---|---|
| Complex protocol/lifecycle FSMs (TCP, PFH) | **Net positive** â€” the reason to use Frame |
| Async hardware-op lifecycles (xHCI: USB enum/transfer) | **Net positive** â€” the TCP "enter kicks the op, completion event advances" shape generalized to register-level hardware unchanged |
| Error/disposition funneling (`=> $^`) | **Net positive** â€” small but repeatable |
| Forced-explicitness â†’ correctness | **Net positive** â€” TCP correct first try |
| Diagrams as committed, checked docs | **Net positive** â€” workflow, not runtime; highest per-system payoff on register-dense protocols (USB) |
| Driver-loop dispatch by `state()` matching | **Mild negative** â€” when an FSM reacts to undifferentiated external completion events, the native driver must match on state names to pick the next event (stringly-typed boundary coupling) |
| Small lifecycle/mode machines | **Neutral** â€” ceremony vs. a `bool`/`enum` |
| Data threading via enter params | **Neutralâ†’negative** â€” vs. function args; doc value only |
| Interrupt-driven paths | **Negative** â€” per-event alloc forces `post`/`drain` |
| Cross-core driving of an FSM (SMP) | **Net positive** â€” the single-owner instance + `post`/`drain` gives cross-core safety for *free* (only `Send` data crosses; no framec `Send`/`Sync` needed). The model's liability becomes the concurrency-correctness mechanism |
| Blocking/diverging control flow | **Negative** â€” forces `PENDING_*` deferral |
| Performance-sensitive hot paths | **Negative** â€” per-event allocation |
| The hard 70% (native primitives) | **N/A** â€” Frame doesn't apply |

- **2026-05-22** â€” **Analysis: is Frame's single-thread-per-instance model a safety advantage?** (Prompted by reflection on the B7/R1 results â€” recorded because it's the load-bearing argument behind the whole OS architecture.) "Can Frame be multi-threaded?" has two answers because it's two questions. **(1) Many instances, one per thread/core â€” yes, proven.** R1a/R1b/R2b drove N independent FSM instances across N cores concurrently with zero locks on FSM state and zero races; it's embarrassingly parallel. **(2) One instance mutated concurrently by many threads â€” no, and deliberately.** framec's generated code is non-`Send`/non-`Sync`; a single instance cannot be driven by multiple threads at once, and you can't recover the capability without destroying the thing â€” a `Mutex<T>` wrapper serializes every dispatch (lock contention, *no* parallelism, plus reentrancy-deadlock risk if a handler dispatches into itself). The reason isn't a missing feature: **a state machine and shared-memory concurrency are semantically incompatible.** Frame's execution model is run-to-completion (RTC) â€” an event, including its transition and enter/exit handlers, is fully processed before the next starts â€” and RTC is precisely what makes "what state am I in?" answerable. Two threads dispatching into one instance produce torn transitions, an exit handler racing the next enter handler, a half-mutated compartment observed mid-flight: the abstraction collapses. This is the actor model (the Erlang backend literally is `gen_statem`: concurrency *between* actors, never *within* one). **Verdict â€” net safety advantage, with one trap.** It makes an entire bug class *unrepresentable*: data races on state, reentrant-dispatch corruption, torn transitions. Rust's `Send`/`Sync` and Frame's RTC jointly *forbid* them â€” you cannot even move a non-`Send` instance across a thread boundary without `unsafe`. The recurring "does framec need `Send`/`Sync`?" gate answered itself across B7/R1: **no â€” because no real systems problem ever needed a shared instance.** Every one decomposed cleanly into per-instance ownership (post/drain for ISRs, single-owner-per-core for SMP, the connection table for TCP). That the constraint never bound in three milestones of systems code is evidence it's the *right* constraint. The deeper, language-design point: RFC-0026's preservation theorem only exists because there's a well-defined operational step (one RTC dispatch) to preserve an invariant across; concurrent intra-instance mutation means there is no atomic "step," so single-ownership is what lets you even *state* soundness. **The trap:** the guarantee only holds if you honor it by message-passing (Send data crosses, instances don't). The moment someone "fixes" the limitation with a `Mutex`, they trade a compile-time guarantee for a runtime lock and get the worst of both. So the honest framing: Frame **trades a capability you almost never want (shared-instance concurrency) for a guarantee you always want (race-free state)**, and the cost only appears if a workload genuinely cannot be sharded â€” which, so far, has never happened.
