# B5 implementation plan — networking (TCP/IP)

> **Status:** Planning. Decisions locked 2026-05-21. See [`../roadmap.md`](../roadmap.md) Track B → B5. "Very large" — the headline milestone and the deepest Frame stress test. Builds on B4's virtqueue + DMA + the `post`/`drain` deferred-event pattern, and B3's per-instance lifecycle + block/wake.

## Goal
A TCP/IP stack — NIC driver, ARP, IPv4, ICMP (ping), UDP, and **TCP modeled as the RFC-793 state machine it canonically is**. The thesis the whole project points at: if Frame expresses a correct TCP connection FSM cleanly, that's the headline result.

## How TCP's requirements map to Frame v4 (the design crux)

Re-reviewed the normative reference (`frame_v4_lang_reference.md` v1.3) + `format_model.md`/`frame_v4_codegen_spec.md`. Frame v4 has exactly 6 constructs (transition, `=> $^`, `push$`, `pop$`, `$.var`, `@@`). The roadmap flagged four "deepest gates"; here's how each actually maps — **none requires a language primitive Frame lacks**:

- **Timers (retransmit, TIME_WAIT 2·MSL, delayed-ACK, keepalive) → enter/exit handlers + state variables + a native timer wheel (post/drain).** Frame has **no** `after(ms)` primitive (confirmed — not in the reference or any v4 doc), and it doesn't need one. The mapping:
  - **State-scoped timers** (e.g. `TIME_WAIT`) arm in the state's **enter handler** `$>()` (`arm_timer(2*MSL)`) and cancel in the **exit handler** `<$()`. This is the natural, idiomatic fit — exactly the use the enter/exit handlers exist for.
  - **Within-state timers** (the retransmit timer in `$Established`, which re-arms per segment without leaving the state) arm/cancel in the **event handlers** (`send`/`ack`), since enter/exit don't fire per segment.
  - **State variables hold the timer bookkeeping** — `$.rto`, `$.retries`, `$.deadline` — and **reset on normal re-entry**, which is exactly "restart the timer fresh each time we enter this state."
  - The actual countdown is a **native timer wheel** (mechanism). On expiry it **posts** a `timeout` event to the connection instance; the kernel **drains** it and fires the Frame event — reusing B4's `post`/`drain` boundary verbatim. Frame owns *when timers are armed/cancelled and what a timeout means in each state*; native owns *the countdown*. Same split as everything else in the kernel.
- **Guards (sequence-number / window predicates) → native `if/else` around transitions.** Frame guards are not a separate construct; they are native conditionals wrapping a `-> $State`, and the framepiler even captures the condition text as a `guard` field for diagram edge labels (`format_model.md` §"Transition Guards from If/Else"). `if in_window($.rcv_nxt, seq) { -> $X } else { /* drop */ }` is a guarded transition. Already the project's idiom (e.g. `PageFaultHandler`'s `is_user ? -> $Killing : -> $Fatal`).
- **History → `push$`/`pop$` (the state/compartment stack).** First-class and tested (`33_doc_history_basic`, `34_doc_history_hsm`); `-> pop$` restores a preserved compartment with state vars intact. RFC-793 doesn't actually require history, so this is available-but-likely-unused for TCP; noted for completeness.
- **Orthogonal / parallel regions → not needed for the connection FSM, and approximated by composition + state vars where useful.** RFC-793 *is* a single FSM, so `TcpConnection` is one HSM, not two parallel regions. The send/receive data-path "independence" is bookkeeping (windows, buffers) held in domain/state vars + native socket buffers, not a Frame parallel region. Where a genuinely independent lifecycle appears (it doesn't for one connection), Frame's answer is **composition** — a parent system holding sub-system instances in its domain (the `Kernel`→`SerialDriver`, `ProcessTable`→`Process`, `Shell`→`Parser` pattern), each advancing on its own events.

**Conclusion:** TCP maps onto Frame v4 without any unsupported primitive. The only thing Frame lacks (`after(ms)`) is supplied by the native timer wheel feeding events through the B4 post/drain boundary — consistent with the project's "Frame owns lifecycle, native owns mechanism" commitment. If a pattern *does* turn out not to map during implementation, we **stop and file a framec bug** rather than work around it (project rule).

## Locked decisions
1. **NIC = virtio-net** (paravirtual, QEMU-native: `-device virtio-net-pci`). Reuses B4's virtqueue + DMA + IRQ machinery; an RX/TX queue pair. e1000 is a later hardware-track option.
2. **Test transport = QEMU user-mode networking (slirp)** now (`-netdev user,...` + host port-forward; no root, CI-friendly). **TAP is the production path and is added to the roadmap as a deferred B5/hardware-track item** (full L2 + true ICMP, but needs host/CI privilege + setup).
3. **Timers = native timer wheel + post/drain**, with Frame owning arm/cancel via enter/exit (state-scoped) and event handlers (within-state) and state vars holding RTO/retries/deadline. **No** reliance on a Frame `after(ms)` primitive (it doesn't exist; not needed).
4. **Guards = native `if/else` around transitions** (framepiler captures the condition for diagrams).
5. **`TcpConnection` is one RFC-793 HSM**, with an `$Open`/`$Active` parent funneling `rst()`/`abort()` to a single `=> $^` disposition → `$Closed` (the `Process.$Alive`/`PageFaultHandler.$FaultActive` pattern). Send/recv windows are domain/state vars, not parallel regions.
6. **Connection demux is native:** a connection table keyed by the 4-tuple `(src ip, src port, dst ip, dst port)` maps an inbound segment to the right `TcpConnection` instance; the drain step posts the segment event to that instance. (The "route the async event to the correct instance" problem, first at scale here.)

## Native / Frame split
- **Native:** virtio-net init (RX/TX virtqueues, DMA buffers, the RX-completion IRQ `post`); Ethernet/ARP/IPv4/ICMP/UDP/TCP header parse + **checksums**; socket send/recv ring buffers; the **timer wheel** (arm/cancel/expire → `post` timeout events); sequence-number/window arithmetic helpers (called from guards); the 4-tuple connection table + demux; the new socket syscalls (`socket`/`bind`/`listen`/`accept`/`connect`/`send`/`recv`/`close`).
- **Frame:**
  - `ArpResolver` — `$Incomplete → $Resolved`, holding a pending-packet queue; ARP-request retransmit via an enter-armed timer; `=> $^` parent for give-up.
  - `TcpConnection` — the RFC-793 HSM: `$Closed → $Listen / $SynSent → $SynReceived → $Established → $FinWait1 / $FinWait2 / $Closing / $TimeWait / $CloseWait / $LastAck → $Closed`, under an `$Active` parent that funnels `rst`/`abort`. Retransmit + `TIME_WAIT` timers via enter/exit + event handlers + state vars; seq/window guards via native `if`. One instance per connection.
  - `UdpSocket` — `$Unbound → $Bound` (datagram in/out; no connection state).
  - `IpReassembly` — fragment reassembly lifecycle (`$Empty → $Reassembling → $Complete | $Timeout`); **may be deferred** if slirp delivers unfragmented datagrams in practice.

## Incremental steps (QEMU-validated each)
1. **virtio-net + RX/TX + `post`/`drain` for frames.** PCI discovery (reuse `pci.rs`), RX/TX virtqueue setup (reuse `virtio_blk` machinery), the RX IRQ `post`s, the kernel `drain`s a received Ethernet frame. Demo: receive + log an inbound ARP request. *The driver-bring-up step.*
2. **ARP + IPv4 + ICMP echo → the kernel answers `ping`.** `ArpResolver` Frame system; native IPv4 + ICMP parse/emit + checksums. (`kernel_answers_ping_b5`, B5-3 — or a TCP-based liveness check if slirp ICMP is unavailable; see Risks.)
3. **UDP + `UdpSocket`.** Bind + echo a UDP datagram against an external client. (Part of B5-5.)
4. **`TcpConnection` (the headline).** RFC-793 HSM; native timer wheel + post/drain for retransmit/`TIME_WAIT`; native socket buffers; guards on seq/window. Handshake → trivial echo/HTTP request-response → clean close, against a real external client (host `nc`/`curl`). (`tcp_echo_or_http_b5`, B5-4.)
5. **IP reassembly** if Step 2/3 show fragmentation in practice (`IpReassembly`). (B5-5.)
6. **Docs + diagrams + CI.** Per-system docs incl. a **`TcpConnection`-vs-RFC-793 comparison write-up** (B5-6); regenerate diagrams; full CI + `qemu-test` (B5-7). Add TAP to the roadmap.

## Expected framec gates / stress findings
- **HSM at scale:** `TcpConnection` is ~11 states + a parent `=> $^` funnel — the largest HSM in the project; tests framepiler HSM codegen at scale.
- **Timed behavior without a primitive:** confirms the enter/exit + state-var + native-wheel + post/drain pattern expresses TCP's timers cleanly — the reference answer for "Frame has no `after(ms)`."
- **Guarded transitions** carrying real predicates (seq/window), with conditions captured for diagrams.
- **Instance scale + async routing:** many concurrent `TcpConnection` instances; the native demux routing drained segment events to the correct instance (the B7 cross-core `post` story rehearsed single-core here).

## Risks
- **virtio-net RX/TX ring + IRQ** — silent hangs, descriptor mgmt. Mitigate: raw frame receive/echo (Step 1) before any protocol logic.
- **TCP correctness** (seq/window/retransmit/`TIME_WAIT`/simultaneous open-close) — the deepest risk. Mitigate: host behavioral tests **per transition** (B5-2), then validate against a real external client.
- **slirp ICMP** — QEMU user-net ICMP echo is restricted on some hosts. Mitigate: if raw `ping` isn't reliable in CI, B5-3 becomes a TCP-handshake liveness check; document the limitation; TAP (roadmap) gives true ICMP.
- **Checksums / offload** — compute in native; don't rely on virtio checksum offload initially.
- **Scope** (5+ sub-steps, each large). Mitigate: commit per step, QEMU-validate each, same rhythm as B3/B4.

## Test mapping
B5-1/B5-2 (`TcpConnection` snapshot + per-transition behavioral, host) → Step 4. B5-3 (ping) → Step 2. B5-4 (TCP handshake + request/response + close) → Step 4. B5-5 (ARP + reassembly) → Steps 2/3/5. B5-6 (docs incl. RFC-793 write-up) → Step 6. B5-7 (diagrams + CI) → throughout.
