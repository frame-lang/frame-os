# B4 implementation plan — block device & filesystem

> **Status:** Active. Decisions locked 2026-05-21. See [`../roadmap.md`](../roadmap.md) Track B → B4. "Very large"; effectively 4 sub-milestones. Builds directly on B3 (preemptive multitasking, `fork`/`exec`/`wait`, the block/wake machinery from Step 5d).

## Goal
Persistent storage and the strongest "same Frame source, host and kernel" demonstration. A block device driver, a buffer cache, a minimal on-disk filesystem (inodes, directories), and a VFS layer — and the H-track `Shell`/`Parser` `.frs` returning as a **userspace program** that reads files and runs programs loaded from disk.

## Locked decisions
1. **Block driver = virtio-blk** (paravirtualized, QEMU-native: `-device virtio-blk-pci`). PCI discovery + a single virtqueue (descriptor table + avail/used rings) in DMA memory + one completion IRQ. AHCI/NVMe is a later hardware-track option.
2. **On-disk FS = a custom minimal inode FS** (xv6-style: superblock, inode table, dirents, free-block bitmap). Formatted + populated by our own **`mkfs` host tool** (a new `xtask` subcommand in Rust). No FAT/ext2.
3. **Userspace shell = scripted** at B4: the shell `.frs` runs in ring 3 and executes a baked command sequence (`cat` a file, run a program from disk) — no interactive console input. A blocking `read`-from-console is a focused follow-on.
4. **The deferred-event `post`/`drain` pattern is born here** (the framec gate): the completion IRQ never dispatches a Frame system. The ISR `post`s a native, interrupt-safe completion record; the kernel `drain`s it from normal context, firing the `BlockRequest` Frame event. Drain reuses B3 Step 5d's block/wake: an I/O-blocked process blocks (`sched::block_current`); drain fires the completion → wakes it.
5. **Disk image** = a second QEMU drive (`virtio-blk`), separate from the Limine ESP boot drive. Built + `mkfs`'d + populated (e.g. with `/bin/hello`, `/motd`) by `xtask` before the smoke run.

## Native/Frame split
- **Native:** PCI config-space scan; virtio-blk init (feature negotiation, virtqueue alloc via `frames` + HHDM, notify/ISR); the completion-IRQ `post` + the kernel-loop `drain`; buffer/page cache (block# → cached buffer, write-back); on-disk FS format (superblock/inode/dirent/bitmap byte layout); VFS dispatch + path lookup; the new syscalls (`open`/`read`/`write`/`close`, `exec`-from-path, file `read`/`write`); the `mkfs` host tool; userspace action implementations for `Shell`/`Parser`.
- **Frame:**
  - `BlockRequest` — `$Queued → $InFlight → $Complete | $Error`. One instance per outstanding I/O; `submit` → `$InFlight`; the drained completion fires `complete`/`fail`. The reference consumer of `post`/`drain`.
  - `OpenFile` — `$Open → $Reading | $Writing → $Closed` (or `$Open ⇄ idle/reading/writing → $Closed`). Per open file descriptor; state-dependent read/write/seek/close.
  - `Mount` — filesystem mount/unmount lifecycle (`$Unmounted → $Mounting → $Mounted → $Unmounting`).
  - `Shell` / `Parser` — **reused H-track `.frs`**, now with userspace (syscall-based) action implementations alongside the existing hosted ones.

## Incremental steps (QEMU-validated each)
1. **virtio-blk + `post`/`drain` + `BlockRequest`** — PCI scan, virtqueue setup, a raw sector read/write. The completion IRQ `post`s; the kernel `drain`s → drives `BlockRequest` → wakes the I/O-blocked initiator. Demo: write a sector, read it back, verify. *The framec-gate step.*
2. **On-disk FS + buffer cache + `Mount`** — define the format; `mkfs` (xtask) formats a disk image; mount it; allocate inodes/blocks; `create → write → read → delete` round-trips by inode through the buffer cache. (`fs_file_roundtrip_b4`, `fs_persists_across_reboot_b4`.)
3. **VFS + `OpenFile` + directories** — path lookup, dirents, the open-file lifecycle; `open`/`read`/`write`/`close` syscalls dispatch through VFS. Read a file by path.
4. **Userspace shell** — build the `Shell`/`Parser` `.frs` for ring 3 with userspace actions; new syscalls for file I/O + `exec`-from-path; the scripted shell `cat`s a file from disk and `fork`/`exec`s a program loaded from disk. (`userspace_shell_runs_program_from_disk_b4`.)

## Expected framec gates
- **The `post`/`drain` deferred-event queue** (interrupt-safe, ideally no-alloc) — the first async-interrupt → Frame boundary; becomes the reference for B5 (NIC) and B7 (cross-core).
- **Same `.frs`, two action backends** at scale — `Shell`/`Parser` building for both host and ring-3 userspace (the headline demonstration).
- Per-inode serialization for concurrent FS ops (multiple processes).

## Risks
- virtio-blk init + virtqueue DMA + the completion IRQ (silent hangs; PCI discovery). Mitigate: validate raw read/write before any FS.
- The `post`/`drain`/wake path crossing interrupt context — must never dispatch Frame from the ISR. Mitigate: native completion record + drain in normal context; lean on the 5d block/wake.
- On-disk format bugs (offsets/endianness) — mitigate with the `mkfs` tool sharing the layout constants and a host-side format check.
- Scope (4 sub-milestones, each large). Mitigate: commit per step, QEMU-validate each, same rhythm as B3.

## Test mapping
B4-1/B4-2 (Frame snapshots + behavioral) → Steps 1–3 (host). B4-3/B4-4 (roundtrip, persistence) → Step 2 (QEMU). B4-5 (userspace shell runs a disk program) → Step 4 (QEMU). B4-6 (`Shell`/`Parser` userspace note + dual build) → Step 4. B4-7 (diagrams, docs, CI) → throughout.
